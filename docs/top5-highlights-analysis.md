## 2. 筛选标准

一条亮点要同时满足四个硬标准，才有资格进前五：

1. **源码真实存在，追问能落到具体文件**。面试官说「打开代码看看」时你能直接翻到。这条淘汰一切「概念性亮点」。
2. **有数字或可复现证据**。「优化了性能」是形容词，「~3MB 降到 KB 级」是证据。数字必须能复现（本项目微基准可用 `RUN_PERF_BENCH=1 npx vitest run src/utils/__tests__/perf.manual.test.ts` 复跑）。
3. **能力信号对准前端实习生 JD**。面试官扫简历只有 30 秒，每条必须能立刻映射到 JD 关键词（性能优化/网络/工程化/复杂业务）。偏后端的能力（RLS、RPC）再好也不主打。
4. **有取舍过程可讲，不是调包完成时态**。「为什么不用 EventSource」「为什么不用加权平均」——有决策链的条目才抗追问；「我用了 xx 库」一追就穿。

外加一个**组合标准**：五条覆盖五个不同维度。两条都讲性能优化，不如一条性能 + 一条工程化——简历是组合拳，不是单项比拼。

## 4. 逐条论证与学习清单

每条按固定结构：解决什么问题 → 为什么亮 → 证据链 → 高频追问 → 翻车点 → 学习清单 → 自测题。**自测题全部能脱口而出，这条才算掌握。**

### 4.1 ① RAG 双路召回 + RRF 融合

**解决什么问题**：单纯向量检索漏专有名词/型号/缩写，单纯关键词检索不懂语义改写；多轮追问「那它怎么部署」直接检索必失败。

**人话版（先读这个）**：用户提问后，系统要先从知识库找出相关文字片段喂给 AI（「召回」）。找法有两种、各有盲区——向量召回比「意思像不像」，懂语义但对 RRF 这类专有名词不敏感；关键词召回是 Ctrl+F 式字面匹配，专有名词必中但「怎么上线」搜不到「如何部署」。所以两路都跑、各取前 8。合并时两路分数量纲不同（像百分制和五分制没法直接加），RRF 干脆不看分数只看名次：总分 = Σ 1/(60+名次)，两路都靠前的片段赢——这就是「免归一化」。找出来的都是垃圾（低于门槛）就不硬塞，让 AI 用通用知识答。分号后是另一个坑：用户追问「那它怎么部署」，AI 看得到历史知道「它」是 Docker，但检索器只拿这一句去搜、「它」搜不出东西——所以检索前先让便宜 LLM 把话补全成「Docker 怎么部署」再搜；补全超过 2.5 秒就放弃、拿原句搜，不让用户干等。

**最大质疑的自洽论证：「检索为什么放前端」（2026-07-10 补）**

先接受事实：通用架构上检索就该在服务端（pgvector），多租户产品把检索放前端是事故。本项目自洽靠四个前提：①规模——几百条切片暴力余弦约 2ms，索引零收益；②设施——无自建后端，Supabase 下服务端方案要么 pgvector（过度设计）要么 Edge Function 每次拉全量算（更慢多一跳）；③数据归属（分水岭）——数据属于用户本人、RLS 只能拉自己的切片，下发本人浏览器无额外暴露面，单用户工具可以、多租户绝对不行；④缓存红利——命中后检索零网络往返。四条件任一不满足即失效、迁 pgvector（有阈值演进设计）。另外澄清：embedding 生成和回答生成都在 Edge Function，前端只承担「检索计算」一段，不是「整个 RAG 在前端」。面试策略：**主动交底**，讲这条时先手带出「规模 + 边界 + 演进」，把最大质疑变成架构判断力展示；求职作品集「把复杂度放可展示层」的动机可半公开为「项目定位是前端主导」，但不能当架构理由讲。

**30 秒口头版（面试被问「讲讲这条」时照这个说）**：知识库问答第一步是找资料。找资料有两种方法——按语义找和按字面找，各有盲区，所以我两路都跑，合并时只按名次不按分数，因为两边分数量纲不一样、没法直接加。找出来的资料质量都不达标时不硬塞给模型，直接降级通用回答，避免垃圾资料误导它。多轮对话里用户会问「那它怎么部署」，「它」是检索不到东西的，所以检索前先用一次轻量模型调用把问题补全，超时或失败就用原句。**简历句是给面试官的压缩电报，口头版才是你要背的——检验标准：合上文档能顺畅说完。**

**RRF 算例（复习时手算一遍就懂）**：积分 = 1/(k+名次)，k=60，两路积分相加排序。设向量路排名 A/B/C，关键词路排名 B/D/A：A=1/61+1/63≈0.0323，B=1/62+1/61≈0.0325，C=1/63≈0.0159，D=1/62≈0.0161 → 最终 B>A>D>C。**两路都靠前的 B 赢**——RRF 奖励的就是「两边都说好」。k 的作用：k=0 时第 1 名积分是第 2 名的两倍（1.0 vs 0.5，榜首独裁）；k=60 时是 0.0164 vs 0.0161（差距平缓，单路榜首不碾压另一路意见）。类比 F1 年度积分：跨场圈速不可比，就按名次换积分再加总。

**为什么亮**：2026 年招前端，AI 应用经验是最强差异化信号，而实习生候选池里 99% 的「AI 项目」只是调 API 画个聊天框。这条证明你理解检索增强的完整工程链路，并且检索代码就跑在浏览器里（`src/utils/` 纯函数）——它是实打实的前端代码，不是「借后端的活充数」。**成色说明**：严格论「前端特有性」，这条是五条里最低的（算法可原样搬去任何运行时），它的定位是「业务纵深记忆点」而非「前端技术证明」——被问「这体现什么前端能力」时落点拉回工程化（纯函数 + 118 单测 + 缓存/Worker 的浏览器侧配套），或诚实分层「纯前端能力主要看流式和存储那几条」。

**证据链**：
- RRF 融合：`src/utils/fuseRetrieval.ts:22`（`DEFAULT_K = 60`，注释里有公式 `score(d) = Σ 1/(k + rank_i(d))`）
- 关键词检索：`src/utils/retrieveChunks.ts`（中文 2-4 gram、密度分、价值判断门槛）
- 查询改写：`src/utils/rewriteQuestion.ts:47`（短问题阈值 12 字）、`:71`（默认 2500ms 超时降级）
- 切片：`src/utils/chunkText.ts:12-13`（300/500，段落→句子→硬切三级）
- 模式分流：`src/views/chat/composables/useChatMessages.ts`（召回不达标降级 general-ai）

**高频追问与答题要点**：
1. *RRF 为什么比加权平均好？* —— 余弦相似度和自定义密度分量纲不同，加权要归一化调参、脆弱；RRF 只用排名，天然免归一化。主动补一句：k=60 是论文经验值，我「设计」的是双路架构和质量门槛，不是算法本身——这句诚实反而加分。
2. *为什么检索放前端不用 pgvector？* —— 阶段性取舍：几百切片量级省一次服务端往返，还能做缓存和 Worker 优化；万级切片应演进 pgvector RPC，按阈值切换。
3. *多轮改写每次都调 LLM 吗？* —— 不。纯函数先判断（有历史 且 问题 ≤12 字或含指代词）才触发，temperature=0、2.5s 超时回退原句；改写只用于检索，展示和落库保留原句。

**翻车点**：说「解决了幻觉」（只能说提升可验证性）；说「自研向量数据库」（embedding 存 Postgres 数组，检索在客户端）；被问 k=60 怎么调出来的时候现编调参过程。

**亮点宣称的是设计，不是位置（2026-07-10 补）**：这条 bullet 通篇宣称的是检索质量设计——双路召回、RRF 秩融合、质量门槛降级、指代改写——这套设计是行业标准方案（Elasticsearch 8 内置 RRF、Azure AI Search 的 hybrid 检索即向量+关键词+RRF、Supabase 官方亦有 hybrid search 指南），写它等于展示「懂标准做法并正确落地」，是选型正确的证据；可疑的只是部署位置（客户端），而 bullet 没有一个字宣称位置，描述行用「浏览器侧」如实标注。「显得选型不对」的印象只在答不出为什么时产生——选型能力不是「选了什么」，是「讲得出约束、理由和失效边界」。若对这条驾驭信心不足，降级排序（流式交互打头、本条退居第二三位）比改写文案更有效；等练透 4.1 再提回第一。

**要学透的知识点**：
- RAG 基本流程：切片 → embedding → 召回 → 注入 Prompt → 生成
- 余弦相似度是什么、为什么能度量语义相近
- 倒排/关键词检索 vs 向量检索的互补性（精确命中 vs 语义泛化）
- RRF 公式含义：k 越大排名差距越平缓
- 为什么切片要控制长度（上下文预算）、无 overlap 的缺点

**自测题**：向量召回和关键词召回各自的失败案例是什么？RRF 公式手写出来？质量门槛不达标时系统做什么？「它怎么部署」这句为什么检索会失败、改写后变成什么样？

### 4.2 ② 手写 SSE 流式解析 + 长回答渲染优化

**解决什么问题**：网络层——OpenAI 兼容接口的 SSE 流需要 POST body 和鉴权头，EventSource 做不到；渲染层——chunk 到达频率高于刷新率、长回答全文重解析会越来越卡。

**人话版（先读这个）**：见[附录 A：亮点②零基础逐坑讲解](#附录-a亮点零基础逐坑讲解2026-07-11)——按「自己从零写会撞上什么坑」的顺序，把简历句里每个词对应的代码讲透。

**为什么亮**：这是五条里**最硬的浏览器基本功**，而且面试官可以零成本现场验证：「中文字符切在半个 chunk 怎么办？」「SSE event 切在半截怎么办？」——答上来就是硬通货，答不上来立刻露馅。这条同时覆盖网络（fetch/ReadableStream/编码）和渲染（rAF/解析成本），一条顶两条。

**证据链**：
- SSE 解析：`src/api/ai.ts`（TextDecoder stream 模式、buffer 按空行切 event、兼容 delta.content/message.content/text、[DONE] 跳过、malformed 忽略）
- rAF 合帧 + AbortController：`src/views/chat/composables/useChatMessages.ts`
- stable/tail 增量渲染：`src/utils/streamingMarkdown.ts`（围栏状态机、未闭合代码块补栏）
- think 分流：`src/utils/streamingThinkParser.ts`（每帧解析累积全文、半截标签隐藏）
- 实测：39k 字符回答、164 次流式切分累计 13ms（≈0.08ms/次）

**高频追问与答题要点**：
1. *为什么不用 EventSource？* —— 只能 GET、不能带 POST body 和 Authorization header；本项目要经 Edge Function 发带参数的 POST。
2. *UTF-8 跨 chunk 怎么处理？* —— `TextDecoder.decode(value, { stream: true })`，stream 模式保留不完整字节序列等下一段补齐。这题考的是「知道多字节字符可能被网络分包切开」这个意识。
3. *rAF 合帧和增量渲染是不是一回事？* —— 不是，两个维度：rAF 控制**更新频率**（一帧最多一次响应式更新），stable/tail 控制**单次渲染量**（稳定段不重复解析）。分得清这两层是加分点。

**翻车点**：说不清 rAF 和增量渲染的区别（最常见）；不知道代码围栏内的空行不能作为段落边界；说「用了 SSE 库」（这条的价值就在手写）。

**要学透的知识点**：
- fetch ReadableStream 的读取循环（`reader.read()` 返回 `{value, done}`）
- UTF-8 变长编码（中文 3 字节）与 TCP 分包无关性
- SSE 协议格式：`data:` 行、空行分隔 event、`[DONE]` 约定
- requestAnimationFrame 与浏览器渲染帧的关系；为什么比节流/防抖更适合这个场景
- Markdown 解析成本随文本长度增长的原因；AbortController 的信号传递链

**自测题**：`decode(value, {stream: true})` 不加 stream 会怎样？一个 SSE event 的完整格式长什么样？为什么 rAF 而不是 `setTimeout(fn, 16)`？停止生成时已输出一半的内容怎么处理（本项目：有内容保留落库，无内容删占位）？

### 4.3 ③ IndexedDB 向量缓存

**解决什么问题**：每次问答重复拉取全量 embedding——500 条 × 1536 维 float 的 JSON 每轮约 3MB 网络传输 + 解析成本。

**人话版（先读这个）**：见[附录 B：亮点③零基础逐坑讲解](#附录-b亮点零基础逐坑讲解2026-07-11)——从「每次提问下载 3MB」出发，把缓存选型、一致性设计、降级兜底逐坑讲透。

**为什么亮**：五条里**量化收益最直观**的一条（~3MB → KB 级，任何面试官秒懂），而且缓存一致性设计能一句话讲清：「切片不可变（重建时整批删旧插新、从不 update），所以 id 集合 diff 就是完整的一致性协议，不需要版本号和 TTL」。观察数据写路径、找到不变量、把复杂问题简化——这是缓存设计的正统思路，讲出来远超实习生水平。

**证据链**：
- diff 算法：`src/utils/embeddingCache.ts:150`（`diffChunkIds` 返回 missingIds/staleIds）
- Float32Array 存储：`embeddingCache.ts:12`（IndexedDB 原生支持 TypedArray）
- missing 过半整表拉取、stale 异步删除、sourceName 不进缓存（可变字段每次新鲜查询）
- 降级：IndexedDB 不可用返回 null，调用方回退全量拉取

**高频追问与答题要点**：
1. *缓存一致性怎么保证？* —— 核心是数据不变量：切片行不可变。服务端只查 id 列表，`serverIds - cachedIds` 增量拉，`cachedIds - serverIds` 淘汰。
2. *为什么不用 localStorage？* —— 同步 API 阻塞主线程、容量小（~5MB）、只能存字符串（序列化 TypedArray 会膨胀且丢结构）。
3. *为什么 Float32Array 不是 number[]？* —— 体积（每元素固定 4 字节 vs JSON 文本十几字节）、IndexedDB 结构化克隆原生支持、后续可直接打包矩阵传 Worker。

**翻车点**：把它吹成「强一致缓存系统」（它依赖不可变假设，假设破坏就失效）；说不出 sourceName 为什么不进缓存（可变字段缓存了会显示旧名）；不知道 IndexedDB 挂了怎么办（回退全量，可用性优先）。

**要学透的知识点**：
- IndexedDB 基本模型：异步、对象仓库、结构化克隆算法能存什么
- localStorage/sessionStorage/IndexedDB/Cache API 的适用边界
- 缓存失效的经典难题与「不可变数据 + 集合 diff」这条捷径
- Float32Array 内存布局（连续 buffer）、与 JSON 数字的体积对比
- ~3MB 怎么算的：500 × 1536 × JSON 文本编码 ≈ MB 级（这是估算口径，面试要能现场推）

**自测题**：missingIds 和 staleIds 分别怎么算？如果切片支持原地 update，这套 diff 还成立吗（不成立，需要版本号/更新时间）？结构化克隆和 JSON.stringify 的区别？

### 4.4 ④ Web Worker + Float32Array 矩阵 + Transferable

**解决什么问题**：向量相似度和关键词评分是 CPU 密集计算；更关键的问题是**跨线程怎么传大数据**——直接 postMessage 传 `number[][]` 会触发大量结构化克隆。

**人话版（先读这个）**：见[附录 C：亮点④零基础逐坑讲解](#附录-c亮点零基础逐坑讲解2026-07-11)——从「主线程为什么怕重计算」讲到「递出去手上变空壳」，并附「实测数字是怎么测出来的」（同时覆盖②的 13ms 测法）。

**为什么亮**：Worker 本身不稀奇，稀奇的是三个细节：把 n 条向量 memcpy 进一个连续 Float32Array 矩阵、只 transfer 一个 ArrayBuffer（复制数据 → 转移所有权）、源缓存向量不 transfer（防 detach，Worker 失败还能主线程重建）。这三个细节是「懂内存模型」的信号。**但叙事必须诚实**：实测 500 条约 2ms，主线程本来不卡——这条讲「为规模增长和低端设备预留余量」，主动报实测数字，展示的是测量意识而不是虚假优化。

**证据链**：
- 矩阵打包 + transfer：`src/utils/retrievalWorkerClient.ts:151-171`（「只 transfer 大头（矩阵，n × dim × 4 字节）」）
- detach 降级：`retrievalWorkerClient.ts:193`（「matrix 可能已被 transfer detach，从源 embeddings 重新打包」）
- Worker 崩溃标记 broken 回退主线程：`retrievalWorkerClient.ts` / `src/workers/retrieval.worker.ts`
- 实测（2026-07-06，Node 中位数）：1536 维矩阵打包+余弦 top8——100 条 ≈1ms、500 条 ≈2ms、1000 条 ≈3ms、2000 条 ≈9ms

**高频追问与答题要点**：
1. *postMessage 传大数组会怎样？* —— 走结构化克隆，深拷贝整个对象图；500 个 number[] 是 500 次数组克隆。打包成一个连续矩阵后 transfer，成本变成 O(1) 的所有权转移。
2. *transfer 之后源数据呢？* —— 源 buffer 被 detach（长度归零不可用）。所以打包的是副本，缓存里的原向量不动；即使矩阵被 detach，也能从源 embeddings 重新打包降级主线程。
3. *Worker 真的解决了卡顿吗？* —— （主动交底）我实测过 500 条约 2ms，当前规模不卡；这个设计的价值是规模余量 + 低端设备 + 把跨线程传输做对。

**翻车点**：吹「解决了主线程卡顿」然后被追问实测数字（0706 定稿明确禁止这个说法）；不知道 detach 是什么；说不出「为什么打包副本而不直接 transfer 缓存向量」。

**要学透的知识点**：
- Worker 通信模型：结构化克隆 vs Transferable objects（哪些类型可 transfer）
- ArrayBuffer / TypedArray / detach 语义
- 行优先矩阵布局：第 i 条向量 = `matrix.subarray(i*dim, (i+1)*dim)`
- 主线程长任务（>50ms）对输入/滚动的影响；为什么 2ms 的任务不值得 Worker 化（除了余量叙事）
- 降级策略设计：性能优化不能成为可用性单点

**自测题**：哪些对象是 Transferable？transfer 一个 buffer 后再读它会发生什么？为什么「复制进矩阵再 transfer」比「500 次克隆」快，明明也复制了一遍（一次连续 memcpy vs 对象图遍历 + 多次分配）？Worker 脚本 404 时用户会感知到什么（无感知，回退主线程）？

## 6. 诚实的局限（学习时心里要有数）

1. **微基准是桌面 Node 环境的中位数**，不是真实浏览器 + 低端手机数据。面试报数字时带上口径（「桌面环境微基准」），反而显得严谨。
2. **~3MB 是估算口径**（500 条 × 1536 维 × JSON 文本编码），不是抓包实测。能现场推出这个数量级即可，不要说成「实测 3MB」。
3. **线上 P50/P95 还没有**：qa_perf 埋点在积累，管理后台图表未做。被问「线上真实数据」就答「埋点已上线在积累，这正是我做分段计时的目的」。
4. **①条对纯 UI 向团队有偏重风险**：如果面试官完全不关心 AI，①的价值靠「这些检索代码都是浏览器里的纯函数 + 118 个单测覆盖」拉回前端视角。

### 6.1 「反正都是调 API」——边界审计（2026-07-10）

针对「向量都是调 API 的，这些亮点会不会名存实亡」的完整审计。**先立正确认知**：embedding 生成和文本生成调 API 是全行业标准（Notion AI、Cursor、各大厂 RAG 无一例外），前端岗没有人期待你自研模型。面试官审的从来不是「模型是不是你的」，而是「**模型之外的东西是不是你的**」。会翻车的只有一种情况：暗示自研——而本项目技术栈里明写「OpenAI-compatible API」，定位话术第一句就是「不是自研大模型」。

| 条目 | API 干的 | 你的代码干的 | 判定 |
| --- | --- | --- | --- |
| ①检索增强 | 文本→向量（/embeddings）、改写那一次 LLM 调用、最终回答生成 | 切片算法、余弦相似度计算、中文 N-gram 关键词评分、RRF 融合、质量门槛与模式分流、改写的触发判断/超时降级架构 | 不虚。「向量召回」中「向量」是 API 产出的数据，「召回」（算分、排序、融合）全是你的代码——用别人的地图数据写自己的导航算法，导航不是虚的 |
| ②流式交互 | 生成的内容本身 | 解析（TextDecoder/事件边界）、中断（AbortController）、渲染（rAF/stable-tail/think 分流）——文案从头到尾没吹内容 | 五条里最纯的前端代码，零依赖模型能力 |
| ③向量缓存 | 被缓存的 embedding 数据 | 整个缓存层：id diff、Float32Array 存储、降级策略 | 不虚。CDN 缓存的图片也不是 CDN 画的 |
| ④计算卸载 | 无 | 余弦相似度是纯数学循环，跑在自己的 Worker；矩阵打包/Transferable/detach 全自己的 | 与 API 零关系 |
| ⑤工程质量 | 无 | 118 用例真实跑过、E2E 真实存在、TTFT 埋点自己埋的 | 不虚 |

**项目里真正「名存实亡」的功能只有一个：AI 写作**（润色/扩写/总结/续写）——纯 prompt 模板 + 调 API，没有工程含量。它只出现在项目描述行的功能列举里、不占 bullet，这个安排是对的；**面试别主动展开讲它的「技术含量」**，被问就答「复用 AI 配置和 Edge Function 代理链路，重点在结果的替换/追加交互」。

**四个真软肋（比「调 API」更可能被戳）及防线**：

1. **「向量召回」没有索引**——是全量暴力余弦，不是 HNSW/IVF。防线：主动交底「几百条 × 1536 维实测 2ms，构建 ANN 索引是负收益；万级才需要索引，那时迁 pgvector 而不是在浏览器造索引」。说清「什么时候不需要索引」比硬上索引更显判断力。
2. **关键词评分不是 BM25**——是简化密度评分（命中率 + 完整命中加分 + 密度/√长度），没有 IDF。防线：「客户端没有全局词频表且切片规模小，简化评分够用且可单测；规模大上 BM25 或服务端全文检索」——是有意简化，不是不知道 BM25。
3. **TTFT 有埋点无线上数字**——被问「你的 P50 是多少」答不出。防线：「埋点已上线在积累，这正是我做分段计时的目的」。
4. **约 3MB 是估算口径**——500 条 × 1536 维 × JSON 文本推出来的，不是抓包。报数字带口径，能现场推导即可。

**收束话术**：这个项目的价值不是「我会调 AI API」，而是 **API 之外的整条工程链路——切片、双路评分、融合、缓存、Worker、SSE 解析、渲染优化、测试——全部是源码级实现**。

### 6.2 「没活找活」焦虑——个人项目的叙事标准（2026-07-10）

针对「这些技术是不是为了简历硬造的、不符合面试官标准」的完整回答。

**先承认事实**，五条按「活的真实性」分三类：

| 分类 | 条目 | 实情 |
| --- | --- | --- |
| 真活 | 流式交互、工程质量 | SSE 高频 chunk、长回答重解析、TTFT 分段是所有 AI 聊天产品的真问题 |
| 派生活 | 向量缓存 | 「每轮拉 3MB」是真问题，但坑来自「检索放前端」的架构选择——自己挖坑自己填 |
| 练习活 | 计算卸载、前端检索本身 | 500 条实测 2ms 不卡，Worker 是刻意练习；个人库大概率到不了需要它的规模 |

**标准校准**：面试官从不用「业务必要性」审实习生个人项目（都知道没真实用户），真实评分维度是三个——真的做出来了吗、知道自己方案的边界吗、诚实吗。「没活找活」在个人项目语境下叫**刻意练习**，不是罪；唯一的红线是**把练习包装成被真实需求逼出来的方案**——被戳穿一条，五条连坐。

**三类活的叙述法**：

- 真活正常讲：问题→方案→实测数字。
- 派生活讲清因果链：「检索放客户端所以 embedding 要下发，才有缓存需求；换 pgvector 这层直接删」——讲得出前提链 = 有架构全局认知。
- 练习活直接承认 + 报实测 + 讲收获：「Worker 是我想练跨线程传输，做完实测 500 条才 2ms——说明这步是提前优化，所以简历只写『预留余量』不写『解决卡顿』，学到性能优化必须先测量。」

## 附录 A：亮点②零基础逐坑讲解（2026-07-11）

> 讲法：**假设自己从零写这个功能，一路会撞上什么坑**——每个坑撞懂了，就懂了那段代码为什么存在。简历句里的每个词就是下面的一个坑：
> 「流式交互：手写 SSE 解析与停止生成，处理中文字符和事件被数据块截断的问题；通过 rAF 合帧和 Markdown 增量渲染减少长回答重复解析，3.9 万字符、164 次切分累计 13ms。」

### A.0 先明白「流式」到底是什么

AI 生成回答是一个字一个字算出来的，生成完一整段可能要 30 秒。如果等全部生成完再显示，用户就盯着空白屏幕干等 30 秒。所以要「生成一个字，屏幕上就多一个字」，像打字机——这就是流式。

服务器往这边发的，是一条**源源不断的文本流**，格式长这样（这个格式就叫 SSE）：

```
data: {"choices":[{"delta":{"content":"今"}}]}
                                              ← 空行，表示「这一条结束了」
data: {"choices":[{"delta":{"content":"天"}}]}

data: [DONE]
```

每条 `data: ...` 里面是一小段 JSON，JSON 里的 `content` 就是新蹦出来的那一两个字。空行是**条与条之间的分隔符**。任务：把这条流里的字一个个抠出来，拼到屏幕上。

浏览器有个现成的 `EventSource` 能自动做这件事，但它**发不了 POST 请求、带不了登录凭证**（Authorization 头），而本项目必须把问题和知识库内容 POST 给 Edge Function——所以用不了，只能用 `fetch` 自己读、自己解析。**「手写 SSE 解析」说的就是这个，被问「为什么手写」答这一段。**

### A.1 第 1 坑：中文变成乱码 �

用 `fetch` 读流，每次 `reader.read()` 拿到的是一小块**字节**（不是字符串），要先把字节翻译成文字。

问题：一个中文字在 UTF-8 里占 **3 个字节**。比如「你」是 `E4 BD A0` 这三个字节。而网络切数据块是**按字节切的，根本不知道字符在哪**，完全可能这么切：

```
第 1 块的结尾: ... E4 BD     ← 「你」的前 2 个字节
第 2 块的开头: A0 ...        ← 「你」的最后 1 个字节
```

如果每块独立翻译，翻译器看到 `E4 BD` 发现「这不是一个完整的字」，只能输出乱码 `�`。屏幕上就会闪出 `今天�`。

**修法一行**（`src/api/ai.ts:234`）：

```ts
decoder.decode(value, { stream: true })
```

`stream: true` 的意思是告诉翻译器：「**后面还有，别急**。」它遇到结尾的半个字，不翻译成 `�`，而是自己先揣兜里，等下一块数据来了，拼上再一起翻译。循环结束后还有一次不带参数的 `decoder.decode()`（`ai.ts:243`）——这是收尾（flush），把兜里可能残留的字节吐干净。

这就是简历里「处理**中文字符**被数据块截断」。

### A.2 第 2 坑：半截 JSON 炸了（含「按空行切」详解）

字节翻译成文字后，要按条解析 JSON。但数据块的切分同样不管「条」的边界，一块数据到手可能长这样：

```
data: {"choices":[{"delta":{"content":"今"}}]}

data: {"cho
```

第一条是完整的，第二条只来了一半。直接 `JSON.parse('{"cho')`——报错。

**修法是「攒着」**（`src/api/ai.ts:234-236`）：

```ts
buffer += 新到的文字               // 全部先倒进一个「攒着」的字符串
const events = buffer.split(空行)  // 按空行切成一条一条
buffer = events.pop()              // ★ 最后一条可能是半截 → 放回去，别处理
// 剩下的每一条都是完整的，放心解析
```

#### 「按空行切」到底是什么意思

其实就是一次普通的字符串 `split()`：

```js
'苹果,香蕉,梨'.split(',')   // → ['苹果', '香蕉', '梨']
```

按逗号切得到三段。「按空行切」是同一件事，只是分隔符从 `,` 换成了「空行」。

「空行」在字符串里长什么样？服务器发来的数据，**眼睛看到**的是三条消息、每条之间空一行；但它在代码里的**真实样子**是一整根字符串，换行只是里面一个看不见的字符 `\n`：

```js
'data: {"content":"今"}\n\ndata: {"content":"天"}\n\ndata: {"content":"好"}\n\n'
```

每条消息结尾有**两个连续的 `\n`**：第一个 `\n` 是「这一行写完了」，第二个 `\n` 是「再空一行」——**连续两个 `\n`，肉眼看过去就是一个空行**。这是 SSE 协议的约定：服务器每说完一条就空一行，意思是「这条完了，下条是新的」，跟写作文段落之间空一行是同一个直觉。

所以「按空行切」翻译成代码就是 `buffer.split('\n\n')`。拿上面那根字符串跑一下：

```js
// → [
//     'data: {"content":"今"}',   ← 第 1 条，完整
//     'data: {"content":"天"}',   ← 第 2 条，完整
//     'data: {"content":"好"}',   ← 第 3 条，完整
//     ''                          ← 结尾 \n\n 后面啥也没有，切出个空串
//   ]
```

#### 那为什么还要 pop？——因为数据是分批到的

上面假装整根字符串一次性到手。实际不是：网络一小块一小块送，切块位置完全随机，可能正好切在一条消息的肚子上。比如第一块到手是：

```js
'data: {"content":"今"}\n\ndata: {"cho'
```

结尾的第二条只传来了 `data: {"cho`，它的后半截和它的 `\n\n` 还在路上。这时 split：

```js
// → [
//     'data: {"content":"今"}',   ← 有 \n\n 收尾，必然完整 ✓
//     'data: {"cho'               ← 最后一段，半截！✗
//   ]
```

规律：**split 结果里，前面每一段都是「以空行收尾」才被切出来的，必然完整；唯独最后一段是「字符串到头了」被迫结束的，可能是半截。** 所以 `events.pop()` 把最后一段拿出来存回 buffer。等下一块数据（比如 `'ices":[...]}\n\n'`）到了，拼到 buffer 后面：

```js
'data: {"cho' + 'ices":[...]}\n\n'  →  'data: {"choices":[...]}\n\n'   ← 接上了！
```

再 split 一次这条就完整了。**每来一块数据就重复：拼进 buffer → 按空行切 → 最后一段留下 → 其余处理。** 半截消息永远只会安静躺在 buffer 里等它的后半身，绝不会被拿去 `JSON.parse` 炸掉。

30 秒验证（浏览器控制台亲手跑）：

```js
'AA\n\nBB\n\nCC'.split('\n\n')          // ['AA', 'BB', 'CC']
'AA\n\nBB\n\nda: {"ch'.split('\n\n')    // ['AA', 'BB', 'da: {"ch'] ← 最后一段是半截
```

注脚：代码里实际写的是 `split(/\r?\n\r?\n/)`（`ai.ts:235`），因为有些服务器的换行是 `\r\n` 两个字符（Windows 风格），`\r?` 表示「有 `\r` 也认，没有也认」，本质不变。这就是「处理**事件**被数据块截断」——「事件」就是一条 `data: ...`。

### A.3 第 3 坑：页面开始卡——更新太频繁

最朴素的显示：每抠出一个字就更新一次 Vue 数据、触发一次页面渲染。

问题：字蹦出来的速度可能是**每秒几十上百次**，而屏幕**每秒只刷新 60 次**。一次刷新的间隔里来了 5 个字，就更新了 5 次数据、跑了 5 次渲染计算——但屏幕只画得出最后那 1 次，**前 4 次全是白干**，白干的活多了主线程就卡。

**修法：攒一帧，画一次**（`src/views/chat/composables/useChatMessages.ts:387-414`）：

```ts
let finalText = ''            // 普通变量，改它不会触发渲染

// 每个字到达时：
finalText += chunk            // ① 只是拼字符串，零成本
if (pendingFrame === null) {  // ② 还没预约「下一次刷新」的话，预约一个
  pendingFrame = requestAnimationFrame(把finalText一次性更新到页面)
}
```

`requestAnimationFrame`（rAF）的意思是：「**浏览器，你下次刷新屏幕之前，叫我一下。**」于是一个刷新间隔内不管来 5 个字还是 20 个字，都只安静拼进 `finalText`，等浏览器要画这一帧了才一次性上屏。5 次白干变成 1 次有效——这就是「**rAF 合帧**」：把一帧内的多次更新**合**并成一次。

类比：快递员每到一个包裹就敲一次门 vs 攒到你正好开门时一起递给你。

为什么用 rAF 而不是 `setTimeout(fn, 16)`：rAF 的回调刚好卡在浏览器下一次绘制前执行，更新必然被这一帧画出来；setTimeout 有漂移，可能在两次绘制之间白跑；且 rAF 在后台标签页自动暂停，不做看不见的更新。

### A.4 第 4 坑：回答越长越卡——每次都「全文重新翻译」

合帧后短回答流畅了，但长回答写到后面又卡。为什么？

页面显示的不是纯文本，是 Markdown 渲染出来的（标题、代码块、加粗）。Markdown 组件的工作方式是：**给它全文，它把全文翻译成 HTML**。于是：

- 回答写到 100 字：每帧翻译 100 字，没感觉；
- 写到 39000 字：**每帧都把 39000 字从头翻译一遍**，而屏幕上实际新增的只有俩字。

浪费在哪：**前面已经写完的段落，内容永远不会再变了，凭什么每帧都重新翻译它？**

**修法：把文章切成「已完稿」和「正在写」两部分**（`src/utils/streamingMarkdown.ts` + `src/views/chat/components/AssistantMarkdown.vue`）：

```
┌────────────────────────────┐
│  stable（已完稿的段落）     │ ← 一个 Markdown 组件渲染它。
│  第一段...                  │    内容没变 → Vue 不会重新渲染它
│  第二段...                  │    → 翻译结果一直复用，零成本
├────────────────────────────┤
│  tail（正在蹦字的最后一段） │ ← 另一个 Markdown 组件渲染它。
│  第三段正在写...█           │    每帧都变，但只有几十个字，翻译很便宜
└────────────────────────────┘
```

页面上其实是**上下两个 Markdown 组件拼起来的**，用户看不出接缝。每帧真正干活的只有下面那个小的。这就是「**Markdown 增量渲染**」——增量 = 只处理新增的部分。

怎么判断「某段已完稿」？**空行**。Markdown 里空行就是段落结束，新字只会往最后一段里加，不会回头改前面。

但有个埋人的陷阱：**代码块里面的空行不算数**。比如 AI 正在输出：

````
```js
function a() {}
                 ← 这个空行在代码块里！
function b() {}
````

要是见到空行就切，代码块会被拦腰切成两半，上半截在 stable、下半截在 tail，渲染直接乱掉。所以 `scanMarkdown`（`streamingMarkdown.ts:41-75`）逐行扫描时一直记着一个状态：「**我现在在不在代码块里？**」（看到 ` ``` ` 进入，再看到 ` ``` ` 出来）——只有**代码块外面的空行**才有资格当切分点。这也是 4.2 翻车点「代码围栏内的空行不能作为段落边界」的出处。

### A.5 最后：停止生成按钮

给 `fetch` 塞一个 `AbortController` 的信号，用户点停止，`abort()` 一喊整条流立刻断掉。断掉后有个贴心分支（`useChatMessages.ts:439-442`）：已经蹦出来的半截回答**保留并存进数据库**；一个字都没出来就**把空气泡撤掉**，当无事发生。注意用户主动停止**不算 error**，不弹错误提示。这就是简历里的「停止生成」。

### A.6 回头读简历那句话（逐词映射）

| 简历用词 | 对应的坑 | 一句话本质 |
| --- | --- | --- |
| 手写 SSE 解析 | A.0 | EventSource 不支持 POST/鉴权头，只能 fetch 自己解析 |
| 停止生成 | A.5 | AbortController；有内容保留落库，无内容撤占位 |

### A.7 面试总纲与自测

**总纲一句话：卡顿有两个原因，分别治**——更新太频繁，用 rAF 合帧治（一帧最多刷一次）；单次更新太重，用 stable/tail 切分治（每帧只翻译最后一小段）。被问「这俩是不是一回事」，答「不是，一个管频率，一个管量」（对应 4.2 高频追问 3）。

**自测**（与 4.2 自测题互补，全部能脱口而出才算过关）：
1. `decode(value, {stream: true})` 不加 `stream: true`，中文会出什么现象？为什么循环结束后还要空参 decode 一次？
2. `events.pop()` 那行删掉会发生什么？（半截 JSON 进 `JSON.parse` 报错）
3. rAF 合帧和 stable/tail 增量渲染是不是一回事？（频率 vs 单次量）
4. 为什么切分点不能用 `lastIndexOf('\n\n')` 一行搞定？（代码块内的空行不是段落边界）
5. 停止生成时已输出一半的内容怎么处理？（有内容保留落库，无内容撤占位）

## 附录 B：亮点③零基础逐坑讲解（2026-07-11）

> 讲法同附录 A：**假设自己从零做这个功能，一路会撞上什么坑**。简历句：
> 「设计 IndexedDB 向量缓存，利用切片不可变的特点按 id 比对增量同步，缓存命中后单轮问答传输由约 3MB 降至 KB 级，不可用时自动回退全量拉取。」

### B.0 先明白这个缓存缓存的是什么东西

用户提问后系统要先从知识库里「找资料」（检索），而且**找资料这一步是在浏览器里算的**（这是亮点①的架构）。找资料靠比较「语义指纹」——每段切片文字都有一个**向量**：一串 1536 个小数，比如 `[0.012, -0.334, 0.078, ...]`，意思相近的文字这串数字就相近。

这些向量存在服务器数据库里。浏览器要算相似度，就必须**先把所有切片的向量拿到手**。问题从这里开始。

### B.1 第 1 坑：用户每提一个问题，都要下载 3MB

最朴素的实现：每次提问 → 从服务器把该知识库全部切片（带向量）拉下来 → 算相似度。

算一笔账：500 条切片 × 每条 1536 个小数 = **768,000 个数字**。它们以 JSON 文本传输——一个小数像 `0.01234567` 写成文字就是十几个字符，再加逗号括号，**3MB 起步**（注意：这是估算口径，500 × 1536 × JSON 文本编码，不是抓包实测，面试报数字带口径）。

用户体验：每问一个问题都要先等 3MB 下载完才开始检索；问 10 个问题，同样的 3MB 下载 10 遍——**而这些向量根本没变过**。

直觉解法：**下载一次，存本地，下次直接用**——「缓存」。听起来一句话，坑全在后面。

### B.2 第 2 坑：存哪？localStorage 装不下

前端存东西第一反应是 localStorage，但它三个硬伤这个场景全踩：

- **容量约 5MB**——一个知识库就 3MB，存两个就爆；
- **只能存字符串**——768,000 个数字得 `JSON.stringify` 成文本，存取都要序列化/反序列化一大坨；
- **同步 API**——读写 3MB 会把主线程卡住，页面冻结。

所以用 **IndexedDB**：浏览器内置的「真数据库」——容量几百 MB 起、API 异步不卡主线程、**能直接存二进制对象**。最后这点配合一个细节（`embeddingCache.ts:12-13` 注释）：向量不存 JSON 数组，存 **Float32Array**——每个数字固定 4 字节，比 JSON 文本省约 75%，读出来不用解析、直接就是能算的格式。（简历 v2 定稿把 Float32Array 从③条里砍了——它在④条已出现，避免关键词重复——但面试被问「为什么不用普通数组」时讲这段。）

### B.3 第 3 坑（这条的灵魂）：我怎么知道本地存的还是新的？

场景：用户昨天缓存了 500 条切片，今天**删了一个文件、又传了一个新文件**——服务器上的切片变了，本地还是旧的。拿旧资料回答新问题 = 答错。这是缓存领域最经典的难题：**缓存一致性**。

教科书解法两种，都不理想：

- **TTL（过期时间）**：「缓存 1 小时后作废重下」。不准——1 小时内照样可能改；没改也白白重下 3MB。
- **版本号**：服务器每次改动版本号 +1，客户端对号。可靠，但要改表结构、每处写入都要维护版本号。

本项目的解法思路不同：**先别急着设计方案，先观察数据是怎么被写入的**。观察发现关键事实：

> 切片这种数据**从来不会被「原地修改」**。重新上传文件时，系统是「整批删掉旧切片、插入全新切片」——新切片拿全新的 id。一条切片从出生到被删，内容一个字不变。

这个特点叫「**不可变**」。它带来威力巨大的推论：

> **只要某个 id 还在服务器上，它对应的内容就绝对没变过。** 判断缓存新旧根本不用比内容——**比一下 id 名单就够了**。

对比网盘同步：网盘除了对文件列表还要比修改时间或内容指纹，**因为文件内容会变**；这里连指纹都不用比，因为规则保证内容永不变。「利用切片不可变的特点」——方案不是拍脑袋，是从数据写入规律里**推导**出来的。

于是每次提问的流程变成（`src/api/chat.ts:590-610`）：

**第一步，向服务器只要一份 id 名单**（不带向量不带内容，几百个 id 就几 KB）：

```
服务器名单: [A, B, C, D, E]
本地缓存:   [A, B, C, X]
```

**第二步，两边名单一比**（`embeddingCache.ts:150` 的 `diffChunkIds`，就是两个 Set 互查）：

```
服务器有、本地没有 → missingIds = [D, E]  → 只下载这 2 条（增量拉取）
本地有、服务器没有 → staleIds  = [X]     → 本地删掉（服务端已删除）
两边都有          → A, B, C             → 直接用，内容不可能变过
```

这就是「**按 id 比对增量同步**」——「增量」= 只补缺的几条，不是全量重下。

**数字兑现**：平时没增没删，名单一比完全一致，一条都不用下载——本轮网络传输只有那份 id 名单，**KB 级**。「缓存命中后单轮问答传输由约 3MB 降至 KB 级」就是这么来的。

### B.4 第 4 坑：真实实现里的四个小坑（`src/api/chat.ts:573-671`）

1. **缺得太多时，一条条补反而慢**。换新电脑缓存全空、500 条全缺——分批补要发好多请求。所以有阈值：**缺失超过一半，直接整表拉一次**（`chat.ts:616`），一个请求搞定。
2. **按 id 批量查有 URL 长度限制**。`.in(id列表)` 是把 id 拼在网址后面发 GET 的，几百个 id 拼上去网址超长报错。所以 **80 个一批**分次请求（`chat.ts:631-633`）。
3. **文件名会变！**「不可变」说的是切片行，但界面显示的「来源：xxx.pdf」文件名是**可变的**（用户可重命名）。文件名进了缓存，重命名后界面永远显示旧名。所以 **sourceName 不进缓存，每次单独新鲜查询**（`embeddingCache.ts:21`）——知道自己「不可变假设」的边界在哪，面试加分细节。
4. **淘汰不着急**。staleIds 的删除**异步扔出去不 await**（`chat.ts:656-657`）——删缓存不影响本轮检索正确性，没必要让用户等。

### B.5 第 5 坑：IndexedDB 挂了怎么办？

IndexedDB 不是永远可用：**无痕/隐私模式**下很多浏览器直接禁用，还有配额满、旧浏览器等。

设计原则一句话：**缓存是加速器，不是必需品——它挂了，功能不能跟着挂**。`embeddingCache.ts` 所有操作静默容错：打不开数据库返回 `null`、写入失败当没写。调用方拿到 `null`（`chat.ts:585-588`）就退回老路——每次全量拉 3MB。**无痕模式下只是慢一点，功能一切正常。**这就是「不可用时自动回退全量拉取」。

### B.6 回头读简历那句话（逐词映射）

| 简历用词 | 对应的坑 | 一句话本质 |
| --- | --- | --- |
| 设计 IndexedDB 向量缓存 | B.1+B.2 | 每轮重复下载 3MB → 存本地；localStorage 三硬伤装不下，用 IndexedDB |
| 利用切片不可变的特点 | B.3 | 切片只整批删旧插新、从不原地改 → id 在 = 内容没变过 |

### B.7 面试总纲与自测

**总纲一句话：缓存一致性不靠 TTL、不靠版本号，靠数据不变量——切片不可变，所以 id 比对就是完整的一致性协议。** 必被追问的一手：「如果切片支持原地修改，这套还成立吗？」答：不成立，id 在但内容可能变了，那时需要版本号或更新时间戳——答得出失效边界，才证明方案是推导的不是抄的。

2. 为什么不用 localStorage？（容量 ~5MB / 只能存字符串 / 同步阻塞主线程）

5. 无痕模式下这个功能会挂吗？（不会，回退全量拉取，只是变慢）

## 附录 C：亮点④零基础逐坑讲解（2026-07-11）

> 讲法同附录 A/B。简历句（定稿）：
> 「将双路检索的评分计算移入 Web Worker；数百条向量合并为单个 Float32Array 整体移交，替代逐条深拷贝；实测 2000 条切片约 9ms，为知识库增长预留主线程余量。」
> C.6 同时讲清 ②「39k 字符 164 次切分 13ms」和本条「2000 条约 9ms」这些数字是怎么测出来的。

### C.0 先明白「主线程」为什么怕重计算

浏览器里一个页面只有**一条主线程**：JS 执行、页面渲染、响应点击滚动打字，全部在这一条线上排队。谁占着不放，页面就冻住。

而检索（亮点①的链路）每次提问都要算一笔重活：问题向量和 500 条切片向量逐条算余弦相似度——500 × 1536 ≈ **77 万次乘加**，外加中文关键词评分。纯 CPU 计算，如果花 200ms，这 200ms 里用户点什么都没反应。

直觉方案：**Web Worker = 浏览器允许你开的一条后台线程**。重活丢过去，主线程该干嘛干嘛，算完把结果送回来。这就是「将双路检索的评分计算移入 Web Worker」。听起来一行 `postMessage` 的事，坑在后面。

### C.1 第 1 坑：数据送过去，默认是「复印」不是「递过去」

Worker 和主线程是**两个隔离的世界，不共享内存**——主线程的变量 Worker 直接看不见（刻意设计，避免两条线程同时改一份数据）。传数据靠 `postMessage`，它默认的行为是：**把数据深拷贝一份**送过去（学名「结构化克隆」）。

具体到这里：500 条向量，每条是装着 1536 个数字的普通数组。直接传，浏览器要逐个遍历、复制 **500 个数组、77 万个数字**——一边想省主线程时间，一边先在主线程干一票复制重活，本末倒置。

类比：要把 500 页资料交给隔壁办公室的同事，默认方式是**全部复印一遍**再送过去。

### C.2 第 2 坑：想直接递原件？可以，但人家只收「一整块」

浏览器提供了不复印的通道：`postMessage` 的**第二个参数**，写进去的数据不复制，直接**把内存所有权移交**给 Worker，瞬间完成、与数据大小无关（学名 Transferable）。

硬条件：**只有一整块连续内存（ArrayBuffer）能这样移交**。500 个普通数组是 500 件散装物品，不是「一块」，不符合资格。

所以先打包（`src/utils/similarity.ts:75-91` 的 `packEmbeddingMatrix`）：

```ts
const matrix = new Float32Array(500 * 1536)   // 开一块约 3MB 的连续内存
// 把 500 条向量一条条抄进去：第 row 条放在 row*1536 开始的位置
```

这就是「**数百条向量合并为单个 Float32Array**」。然后（`src/utils/retrievalWorkerClient.ts:173-183`）：

```ts
postMessage({ ...请求, matrix }, [matrix.buffer])
//                              ↑ 第二个参数一指定：这 3MB 瞬间归 Worker，零复制
```

这就是「**整体移交，替代逐条深拷贝**」。类比接上：把 500 页散纸**装订成一卷档案**再整卷递过去——装订（打包）是自己做的一次紧凑复制，递交那一下零成本。

见功力的小细节（`retrievalWorkerClient.ts:171-172` 注释）：**问题向量没有一起移交**，走默认克隆——它只有一条、6KB，克隆成本忽略不计，还保住了调用方手里的原件。只移交「大头」，是权衡不是无脑全移。

### C.3 第 3 坑：递出去之后，你手上的变成空壳

移交所有权是真的交出去了：transfer 之后**主线程这边的 matrix 立刻报废**——长度归零、不可再用（学名 detach）。

埋伏：万一 Worker 挂了要退回主线程算——**拿什么算？matrix 已经是空壳了！**

解法（`retrievalWorkerClient.ts:193-200`）：降级路径**从源向量重新打包一次**再算。能重新打包，是因为打包进矩阵的从头就是**副本**——IndexedDB 缓存里的源向量从没被移交过，一直安然无恙。这就是面试必问「为什么打包副本、不直接移交缓存里的向量」的答案：原件交出去就没了，下次提问缓存就废了。

### C.4 第 4 坑：Worker 本身也会挂——降级兜底

Worker 脚本可能加载失败（旧浏览器、构建产物路径问题），运行时也可能崩。处理与③的降级同一哲学——**性能优化不能成为可用性单点**（`retrievalWorkerClient.ts:73-79`）：

- 脚本级错误 → 打 `workerBroken` 标记，在途请求全部转走，之后的调用不再尝试 Worker，直接主线程同步算；
- 单次调用失败 → `console.warn` 一声，当次降级主线程。

用户全程无感知，最坏情况只是「没享受到后台计算」。另外两个实现点，被问「Worker 怎么管理」时用：**懒加载单例**（首次调用才创建）；**自增 id 对号入座**（`retrievalWorkerClient.ts:100-106`）——并发多个请求各带编号，Worker 回话凭编号找到对应 Promise，不串线。

### C.5 第 5 坑（反转）：测完发现——当前规模根本不卡

微基准结果（2026-07-06，Node 中位数）：**500 条约 2ms，2000 条约 9ms**。一帧的预算是 16.7ms——当前几百条的规模，主线程算也不卡。

所以简历收尾写「**为知识库增长预留主线程余量**」，不写「解决了主线程卡顿」——后者会被自己的实测数字当场拆穿（4.4 翻车点）。被问「这优化值得吗」，用 6.2 节的王炸反思链：「**做了 → 测了 → 发现当前规模不需要 → 叙事修正为预留余量**，这件事教会我性能优化必须先测量。」敢承认过度优化，比硬吹优化更值钱。

### C.6 数字是怎么测出来的（`src/utils/__tests__/perf.manual.test.ts`）

**测耗时的本质 = 掐秒表**，和百米计时一样：起跑看表，冲线看表，相减即用时。代码里「看表」就是 `performance.now()`（返回当前时刻，精确到零点几毫秒）：

```js
const t0 = performance.now()   // 起跑：表读数 1000.0
算一遍2000条的相似度()          // 跑
const t1 = performance.now()   // 冲线：表读数 1009.2
console.log(t1 - t0)           // 9.2 毫秒 —— 测完了
```

所有性能测量的核心就这三行。围绕它有四个配套设计：

1. **数据是假的，耗时是真的**。测试不连数据库，用固定种子的伪随机造 2000 条假向量——对 CPU 来说 `0.7×0.3` 和 `0.5×0.9` 一样快，**耗时只取决于算多少次，不取决于数字真假**；固定种子保证每次跑数据相同、结果可对比（这也是代码里不用 `Math.random()` 的原因）。
2. **跑 5 次取中位数，不取平均**。电脑掐表瞬间可能在干别的（如垃圾回收），5 次成绩可能是 `[9.1, 9.3, 8.9, 22.7, 9.0]`——平均 11.8 被那次 22.7 骗了；排序取中间 `[8.9, 9.0, →9.1←, 9.3, 22.7]` = 9.1，怪异值挤在两端影响不到中间。
3. **分段计时 + 阶梯规模**。打包耗时和计算耗时分开记（知道成本花在抄数据还是算数）；100/500/1000/2000 四档看**增长趋势**（近线性）——「涨几倍也有余量」的外推靠阶梯数据支撑。
4. **环境变量开关**。`RUN_PERF_BENCH=1` 才执行，平时 `npm test` 自动跳过（测的是「多快」不是「对不对」，不让 CI 陪跑）。复现命令：

```
RUN_PERF_BENCH=1 npx vitest run src/utils/__tests__/perf.manual.test.ts
```

跑完 `console.table` 打出成绩表，「2000 条约 9ms」就是 total 中位数那一格。**面试前亲手跑一遍，报数字有底气。**

**②的 13ms 同一个文件测的**，多一个巧思——怎么模拟「字一帧一帧到达」？把拼好的 3.9 万字全文**每 240 字符切一刀**，得 164 个「进度快照」（快照 1 = 前 240 字，快照 2 = 前 480 字……快照 164 = 全文）。真实流式里切分器每帧面对的就是「累积全文」，这 164 个快照就是 164 帧的真实工作量；全部跑一遍 `splitStreamingMarkdown` 计总时长 ≈ 13ms。

**口径必须会讲**：这是**桌面 Node 环境的微基准、中位数**——引擎与 Chrome 同款（V8），数量级可信，但不是真机浏览器 + 低端手机数据（6.1 诚实局限第一条）。报数字带口径反而显严谨。**能这么测的前提是被测函数全是纯函数**（不碰网络/DOM）——这是⑤条「核心算法抽成纯函数」的红利：能测对错，也能测快慢。

| --- | --- | --- |

### C.8 面试总纲与自测

**总纲一句话**：跨线程传大数据的关键是「把散装变一整块，再移交所有权替代复制」；而这条真正的面试价值在反思链——**做了、测了、发现当前不需要、所以只叫预留余量**。

**自测**（与 4.4 自测题互补，全部能脱口而出才算过关）：
1. Worker 和主线程为什么不能直接共用一个变量？（内存隔离，通信只能靠消息）
2. 直接 `postMessage` 传 500 个普通数组会发生什么？（结构化克隆，逐个深拷贝）
3. 为什么必须先合并成一个 Float32Array 才能「整体移交」？（移交只认一整块连续内存）
4. 移交后主线程再读 matrix 会怎样？降级时拿什么算？（空壳/detach；从没被移交过的源向量重新打包）
5. Worker 脚本 404，用户会感知到什么？（无感知，broken 标记 + 主线程兜底）
6. 「500 条才 2ms，这 Worker 白做了？」怎么答？（做了→测了→发现不需要→修正叙事为预留余量）
7. 9ms 和 13ms 这些数字怎么来的？（带开关的 vitest 微基准：固定种子假数据、5 遍取中位数、分段计时；桌面 Node 口径）
