## **第 1 问：AI 问答检索链路**

你简历里写到“基于文本切片、Embedding 余弦相似度检索与关键词检索等，封装动态切换问答模式”。

请你完整讲一下：用户在前端输入一个问题后，从“问题提交”到“AI 开始流式返回第一个 token”之间，你们前端和 Supabase/后端分别做了哪些事情？

我会重点听这几件事：  
`切片如何命中`、`向量相似度怎么计算/排序`、`关键词检索怎么和向量检索融合`、`动态切换问答模式的依据是什么`、以及`为什么这些逻辑放在前端或后端是合理的`。


你至少要能讲清楚一条链路：

用户输入问题后，前端先校验登录态、知识库 ID、当前问答模式，然后把问题、知识库范围、历史上下文发送给服务端。服务端会对问题生成 query embedding，再去 Supabase 的向量表里做相似度检索，通常用 cosine similarity 或 inner product，拿 topK 片段。同时可能做关键词检索，比如对标题、正文、标签做 `ilike` 或全文搜索。最后将向量结果和关键词结果合并、去重、重排，把命中的 chunk 拼成 prompt context，再调用大模型接口，并通过 SSE 把 token 流式返回给前端。

如果你要体现前端价值，要补一句：前端不是只展示结果，而是负责“问答模式抽象”和“交互体验控制”。比如普通问答、知识库问答、当前文档问答三种模式，对应不同的检索范围、请求参数、空结果兜底和 UI 状态。

## **第 2 问：追问流式对话**

你简历写了“实现流式对话体验，并解析数据流逐段更新 AI 消息并在完成后持久化到 Supabase”。

那我问具体一点：

SSE 返回的数据流可能被浏览器拆成不完整的 chunk，比如一次只返回半个 JSON 字符串。你前端是怎么解析流的？如果中途网络断开、用户切换页面、或者回答还没结束就刷新页面，你怎么保证 UI 状态和数据库里的消息状态不会乱？


你可以这样回答：

我实现流式对话时，核心不是等完整响应回来，而是边接收边渲染。
前端会把 AI 回复先创建成一条“生成中”的临时消息，然后持续读取服务端推过来的 SSE 数据。因为流数据可能被拆包，所以我不会每次收到数据就直接解析 JSON，而是维护一个缓存区，把不完整的数据先拼起来，等识别到完整的 data: 行或结束标记后再解析。
每解析出一段内容，就追加到当前 AI 消息里；收到结束标记后，把消息状态改成完成，并统一写入 Supabase。
如果中途断网或用户离开页面，就把状态标记为失败或中断，保留已生成内容，但不会把它当作完整回答处理，避免数据库里出现半截但状态正常的消息。


第 3 问：文件切片为什么放前端？怎么切？

答案：

我们把部分切片逻辑放在前端，主要是为了降低服务端压力，也让上传过程可控。
切片不是简单按固定字数粗暴截断，而是优先按段落切，再按句子兜底。如果一个段落过长，再按最大长度拆分，并保留一定 overlap，避免上下文被切断。
每个 chunk 会带上文档 ID、用户 ID、序号、原文内容、标题等元信息，后续检索命中后可以回溯到原文档。
批量入库时不会一次性全塞，而是分批写入，避免 Supabase 请求过大或某一批失败导致整篇文档全部失败。失败批次可以重试。

关键词：段落优先、句子兜底、最大长度、overlap、元信息、分批入库。


第 4 问：向量检索和关键词检索怎么融合？

答案：

向量检索适合找语义相近的内容，比如用户问法和原文不完全一致；关键词检索适合精确词、专有名词、标题、编号这种场景。
所以我不会只依赖一种检索。一般会先根据问题生成向量，拿到 topK 语义相似 chunk；同时对标题、正文或标签做关键词匹配。
两路结果合并后去重，再根据相似度分数、关键词命中、文档更新时间或当前文档优先级做重排。最后只选最相关的几个 chunk 放进 prompt，避免上下文太长影响模型质量和成本。
如果向量结果很弱，但关键词命中很强，就偏关键词；如果用户问的是总结、解释类问题，就偏向量检索。

关键词：向量找语义、关键词找精确、合并去重、重排、topK、控制 prompt 长度。


第 5 问：动态切换问答模式到底动态在哪？

答案：

我这里的动态切换不是 UI 上换个 tab，而是请求参数、检索范围和兜底策略不同。
比如普通问答不走知识库，只带历史上下文；知识库问答会限定用户选中的知识库范围；当前文档问答只检索当前文档的 chunk。
前端会把这些模式抽象成统一的配置，比如 mode、knowledgeBaseId、documentId、是否启用关键词检索、topK 数量等。
这样 UI 层只关心用户选择，业务层根据模式生成不同请求，后面新增“团队空间问答”或“共享文档问答”也比较容易扩展。

关键词：不是 tab，是检索范围和策略变化；模式配置化；扩展性。


第 6 问：你说优化长回复响应感知，怎么衡量？

答案：

我不会说“感觉更快了”，我会拆两个指标。
一个是首 token 时间，也就是用户点击发送到第一段 AI 内容出现的时间。另一个是完整回答耗时，也就是整条回复结束的时间。
流式优化主要改善的是首 token 时间和用户感知，而不是一定减少完整生成时间。
原来如果等接口完整返回，用户可能 10 秒都看不到东西；改成流式后，可能 1 到 2 秒就看到第一段内容。即使总耗时还是 10 秒，用户也会觉得系统在响应。
前端可以通过发送时打点、首段内容渲染时打点、完成时打点，上报到后台看板。

关键词：首 token 时间、完整耗时、感知性能、打点验证。


第 7 问：文档自动保存和本地草稿怎么避免冲突？

答案：

我把它分成三层：防抖保存、本地草稿、远端版本。
用户编辑时不会每个字符都请求，而是防抖后保存，减少频繁写库。保存前会记录保存状态，比如保存中、已保存、失败。
如果网络失败，内容先存在本地草稿里，用户刷新后可以恢复，避免数据丢失。
冲突处理上，至少要比较本地更新时间和远端更新时间。如果远端版本更新，不能直接覆盖，需要提示用户选择使用本地草稿还是远端版本。
对当前文档同步知识库切片时，也不能每次编辑都同步，而是用户明确触发，或者保存稳定后再同步，避免知识库里出现大量中间态内容。

关键词：防抖、保存状态、本地草稿、更新时间比较、避免中间态切片。


第 8 问：Pinia 登录态和权限控制怎么做才可靠？

答案：

Pinia 主要管理用户信息、session、角色和权限状态。页面刷新后，需要从 Supabase 恢复 session，而不是只依赖内存状态。
路由守卫里会判断两件事：是否登录，以及是否有 admin 权限。没有登录跳登录页，没有权限跳无权限页。
但我会强调：前端权限只能做体验控制，不能当安全边界。真正的数据权限必须由 Supabase 的 RLS 或后端校验保证。否则用户绕过路由，直接调接口还是可能拿到数据。
所以前端负责“能不能看到入口”，后端或数据库策略负责“能不能真正访问数据”。

关键词：刷新恢复 session、路由守卫、角色判断、前端不是安全边界、RLS 才是关键。
