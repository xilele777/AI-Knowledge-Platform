# 项目拷打问答

## 1. 你这个项目为什么选择 Supabase BaaS，而不是自己写后端？

这个项目本质是一个 AI 知识平台，核心目标是快速打通“用户认证、文档管理、知识库、问答、分享、管理后台”这条产品链路。Supabase 提供了 Auth、Postgres、RLS、RPC 和实时可扩展的数据库能力，所以在项目早期用它可以减少后端样板代码，把主要精力放在前端业务闭环和 AI/RAG 体验上。

代码上，前端通过 [supabase.ts](f:/AI-Knowledge-Platform/src/utils/supabase.ts) 初始化 Supabase Client，具体业务访问封装在 [api](f:/AI-Knowledge-Platform/src/api) 目录里，例如 [documents.ts](f:/AI-Knowledge-Platform/src/api/documents.ts)、[knowledge.ts](f:/AI-Knowledge-Platform/src/api/knowledge.ts)、[chat.ts](f:/AI-Knowledge-Platform/src/api/chat.ts)。这样页面层不会直接散落数据库查询，而是通过 API 层拿到统一的 `success/data/error` 结构。

设计选择上，这种方式适合 MVP 和中小规模应用：开发快、部署简单、认证和数据库权限能用 RLS 兜底。代价是部分复杂逻辑会落在前端，比如文档切片、AI 调用、同步知识库等。后续如果要生产化，我会把高风险或重计算逻辑迁移到 Edge Function 或自建后端，比如 AI Key 代理、文档同步事务、向量检索、管理员跨用户操作。

## 2. 前端直连 Supabase，安全边界到底在哪里？

安全边界不在前端。前端的 Vue Router 守卫、按钮显隐、`owner_id` 查询条件都只是体验层和调用约束，不能作为最终安全保证。真正的安全边界是 Supabase Auth + Postgres RLS。

项目的 SQL 里已经对 `documents`、`knowledge_bases`、`knowledge_files`、`knowledge_chunks`、`knowledge_documents`、`chats`、`chat_messages`、`analytics_events`、`user_ai_config` 等表启用了 RLS，并通过 `auth.uid() = owner_id/user_id` 限制数据归属。前端 API 里也会先通过 `getCurrentUser()` 获取当前用户，再在查询里追加 `owner_id = userId`，例如 [chat.ts](f:/AI-Knowledge-Platform/src/api/chat.ts) 的会话、消息查询，以及 [documents.ts](f:/AI-Knowledge-Platform/src/api/documents.ts) 的文档查询。

这个设计的原则是“双层约束”：前端过滤让业务语义更明确，数据库 RLS 防止恶意请求越权。改进上，我会把管理员类接口和 AI Key 相关接口迁移到 RPC/Edge Function，并在服务端再次校验 admin 权限，减少前端直连普通表带来的风险。

## 3. 用户的 AI API Key 存在 Supabase 表里，是否有泄漏风险？

有风险。当前项目支持用户自带 OpenAI 兼容服务，所以 [userAiConfig.ts](f:/AI-Knowledge-Platform/src/api/userAiConfig.ts) 会从 `user_ai_config` 表读取 `api_key`，然后浏览器直接带着这个 Key 请求用户配置的 AI 服务。

这个设计的优点是平台不绑定单一模型供应商，用户可以自定义 `baseUrl / apiKey / model`，平台自身也不承担模型调用成本。但它的缺点很明确：Key 会进入浏览器运行环境，也会出现在浏览器发出的网络请求里。RLS 只能保证其他用户读不到这条配置，不能解决“当前用户浏览器可见明文 Key”的问题。

更安全的生产方案是：前端永远不读取明文 Key；Key 在数据库中加密存储；AI 请求统一走 Edge Function 或服务端代理；服务端读取 Key、转发请求、做限流、审计、错误脱敏和 token 统计。也就是说，关键不是“数据库里是否加密”这一点，而是不要让浏览器侧拿到明文 Key。

## 4. 你的 RAG 流程到底是什么？不是简单把全文塞给模型吗？

不是全文塞给模型。项目的 RAG 链路分成四步：内容入库、检索召回、Prompt 构建、流式生成。

内容入库阶段，文档或 txt/md 文件会经过 [chunkText.ts](f:/AI-Knowledge-Platform/src/utils/chunkText.ts) 切成 300 到 500 字左右的片段，然后写入 `knowledge_chunks`。上传文件走 [FileUpload.vue](f:/AI-Knowledge-Platform/src/components/knowledge/FileUpload.vue)，文档加入知识库走 [DocEditorView.vue](f:/AI-Knowledge-Platform/src/views/docs/DocEditorView.vue) 和 [documents.ts](f:/AI-Knowledge-Platform/src/api/documents.ts) 的 `addDocumentToKnowledgeBase()`。

检索阶段在 [ChatView.vue](f:/AI-Knowledge-Platform/src/views/chat/ChatView.vue) 的 `prepareSmartQa()` 里完成。它会拉取当前知识库最多 500 个切片，优先尝试 embedding 向量召回；如果没有 embedding、AI 配置不完整、embedding API 失败，或者向量结果质量不够，就回退到 [retrieveChunks.ts](f:/AI-Knowledge-Platform/src/utils/retrieveChunks.ts) 的关键词 + 中文 ngram 检索。

Prompt 构建阶段由 [buildQaPrompt.ts](f:/AI-Knowledge-Platform/src/utils/buildQaPrompt.ts) 完成，会把命中的知识片段、用户问题、系统指令和回答风格拼接成模型输入。最后由 [ai.ts](f:/AI-Knowledge-Platform/src/api/ai.ts) 通过 SSE 流式生成回答，并把回答和引用 sources 保存到 `chat_messages`。这样不是“全文塞模型”，而是先召回相关片段，再把 TopK 上下文注入模型。

## 5. Embedding 在项目里到底有没有真正用起来？

已经用起来了。文件上传时，[FileUpload.vue](f:/AI-Knowledge-Platform/src/components/knowledge/FileUpload.vue) 会在用户 AI 配置完整时调用 `batchInsertKnowledgeChunks(..., { generateEmbeddings: true })`，由 [knowledge.ts](f:/AI-Knowledge-Platform/src/api/knowledge.ts) 里的 `batchInsertKnowledgeChunks()` 批量生成 embedding 并写入 `knowledge_chunks.embedding`。

文档加入知识库时，[DocEditorView.vue](f:/AI-Knowledge-Platform/src/views/docs/DocEditorView.vue) 会把当前 AI 配置传给 `addDocumentToKnowledgeBase()`，如果配置完整，也会为文档切片生成 embedding。问答时，`prepareSmartQa()` 会为用户问题调用 [vectorEmbedding.ts](f:/AI-Knowledge-Platform/src/utils/vectorEmbedding.ts) 的 `createEmbedding()`，再用 `findTopSimilarChunks()` 做余弦相似度排序。

设计上是“向量优先 + 关键词兜底”。如果切片有 embedding，就做语义召回；如果没有 embedding 或者召回质量不够，就走关键词检索。后续优化方向是把向量检索从前端内存计算迁移到数据库 pgvector 或服务端向量索引，并把聊天模型和 embedding 模型拆开配置，避免供应商模型兼容问题。

## 6. 为什么还要做关键词 fallback？向量检索不是更好吗？

向量检索更适合语义召回，但它不是永远可用。它依赖几个前提：切片必须已经有 embedding；用户 AI 配置必须完整；embedding API 必须兼容并可用；返回向量维度必须一致；召回分数也要足够可信。

项目里很多情况下这些前提不成立。比如用户没有配置 AI Key，历史切片没有生成 embedding，embedding 接口限流失败，或者供应商虽然兼容 chat completions 但不兼容 embeddings。为了保证核心问答功能不因为 AI 能力失败而完全瘫痪，`prepareSmartQa()` 会在向量召回失败或质量不足时回退到关键词检索。

关键词检索在 [retrieveChunks.ts](f:/AI-Knowledge-Platform/src/utils/retrieveChunks.ts) 里实现，会对英文做 token，对中文做 2 到 4 字 ngram，并结合命中比例、命中密度、问题精确匹配进行打分。它不如向量检索智能，但实现轻量、稳定、可解释，适合作为兜底方案。

## 7. 文本切片为什么按字符长度，而不是 token？会有什么问题？

当前 [chunkText.ts](f:/AI-Knowledge-Platform/src/utils/chunkText.ts) 是按字符长度近似控制切片大小。逻辑是先按段落拆分，再按句子拆分，超长文本硬切，最后把过短片段合并到 300 到 500 字左右。

这样做的优点是实现简单、无额外依赖、速度快，在中文文档场景下可以快速落地。缺点是字符数不等于模型 token 数，尤其是中英文混排、Markdown 表格、代码块时，字符长度和实际 token 消耗会有偏差。它也可能切断标题、列表、代码块等语义结构。

如果继续优化，我会引入 tokenizer，按 token 上限切片；用 Markdown AST 或标题层级做结构化切片；给相邻切片增加 overlap；同时在 chunk meta 里保留标题路径、来源文档、段落位置等信息，提高召回后的可解释性。

## 8. SSE 流式回答是怎么实现的？如何处理半包和不同模型格式？

[generateAiTextStream](f:/AI-Knowledge-Platform/src/api/ai.ts) 使用 `fetch` 请求 OpenAI 兼容接口，并通过 `response.body.getReader()` 读取 `ReadableStream`。每次读到二进制 chunk 后，用 `TextDecoder.decode(value, { stream: true })` 增量解码，这样可以处理中文多字节字符跨 chunk 的情况。

解析时维护一个 `buffer`，按 SSE 协议的空行分隔事件块，再解析每个事件里的 `data:` 行。因为不同 OpenAI 兼容服务返回结构不完全一致，所以代码兼容了 `delta.content`、`message.content`、`text` 等多种字段。解析到增量文本后，通过 `onChunk` 回调实时更新聊天界面的本地 assistant 消息。

如果请求失败或断流，函数会返回失败结果，聊天页会把消息状态置为 `error`。如果已经收到部分文本，会保留部分文本，避免用户看到完全空白。后续可以加显式停止生成、超时控制、重试机制、服务端 trace id，以及服务端流代理，提升可观测性和稳定性。

## 9. 聊天消息保存失败怎么办？用户已经看到 AI 回答了，数据不一致如何处理？

当前链路是：先保存用户消息，再创建一条本地 `streaming` 状态的 assistant 消息。SSE chunk 回来时持续更新这条本地消息；流结束后，`persistAssistantFinal()` 把完整回答、sources、answerMode、status、errorMessage 保存到 `chat_messages`。

如果 AI 生成失败，项目会保存一条 `status=error` 的 assistant 消息，避免完全没有记录。但有一个边界情况：如果 AI 流式生成成功，用户已经看到了答案，最后保存 assistant 消息失败，就会出现前端显示和数据库记录不一致。

更稳的方案是生成前先在数据库插入一条真实的 `status=streaming` assistant 消息，拿到真实 messageId；流式过程中节流更新这条记录；结束后改为 `done`；失败则改为 `error`。同时给请求加 client request id 做幂等，避免网络重试造成重复消息。

## 10. 路由权限怎么做？刷新页面时会不会误跳登录页？

[main.ts](f:/AI-Knowledge-Platform/src/main.ts) 在注册 Router 前会先 `await userStore.initialize()`，从 Supabase 恢复当前 session。[router/index.ts](f:/AI-Knowledge-Platform/src/router/index.ts) 的 `beforeEach` 里也有兜底逻辑：如果用户 store 还没初始化，会再次初始化。

这样刷新页面时，不会在 session 还没恢复时就误判为未登录并跳到 `/login`。路由守卫会处理三类情况：未登录访问业务页跳登录；已登录访问登录/注册页跳业务页；非管理员访问 `/admin` 跳回普通页面。

管理员角色由 [user.ts](f:/AI-Knowledge-Platform/src/stores/user.ts) 的 `resolveRole()` 解析，来源包括 `app_metadata.role`、`user_metadata.role`、`app_metadata.is_admin`、`user_metadata.is_admin`。但我会强调，前端路由权限只是体验层，真正的数据权限仍然要靠 RLS、RPC 内部 admin 校验或后端接口。

## 11. 管理后台统计和管理操作是怎么来的？会不会被 RLS 卡住？

管理后台有两类能力。一类是统计概览和用户列表，项目里更倾向使用 Supabase RPC。例如 [admin.ts](f:/AI-Knowledge-Platform/src/api/admin.ts) 的 `getAdminAnalyticsOverview()` 调用 `admin_get_analytics_overview`，RPC 内部做 admin 校验后再聚合用户数、文档数、AI 调用趋势等数据；用户列表也优先走 `admin_get_profiles`。

另一类是普通表的跨用户查询和删除，比如 `getAdminDocuments()`、`getAdminKnowledgeFiles()`、`getAdminChatRecords()`、`adminDeleteChat()`。这些目前仍是前端直接查/删普通表。如果数据库只配置了 owner RLS，而没有 admin policy 或安全定义者 RPC，这些操作在真实环境可能被 RLS 卡住，导致管理员拿不到全量数据或删除失败。

所以生产化时，我会把所有跨用户查询、删除、统计都收敛到 admin RPC 或 Edge Function。前端只调用受控接口，服务端统一校验 admin 身份，避免前端直接对高权限表操作。

## 12. 为什么用户状态和 AI 配置都放 Pinia？有没有过度全局化？

用户状态和 AI 配置都是跨页面共享状态，所以放 Pinia 是合理的。[user.ts](f:/AI-Knowledge-Platform/src/stores/user.ts) 管理 session、user、role、登录状态，路由守卫、布局、后台入口都依赖它。

[aiConfig.ts](f:/AI-Knowledge-Platform/src/stores/aiConfig.ts) 管理用户 AI 配置，文档 AI 助手、知识库向量化、聊天问答都需要它。如果每个页面各自加载配置，会造成重复请求、状态不一致，也不利于统一判断 `isComplete` 和缺失字段。

但项目没有把所有状态都塞进 Pinia。像聊天页的抽屉开关、输入 loading、当前消息列表、文档编辑器表单、弹窗状态，这些页面临时态仍然留在组件内部。我的原则是：跨页面共享、影响权限或全局能力的状态进 store；页面局部交互态留组件。

## 13. 文档编辑器如何避免用户丢内容？保存成功后草稿是否清理？

项目有 [documentDraft.ts](f:/AI-Knowledge-Platform/src/utils/documentDraft.ts)，用 `localStorage` 按 `doc_draft_v1:{documentId}` 保存标题、内容和更新时间。

[DocEditorView.vue](f:/AI-Knowledge-Platform/src/views/docs/DocEditorView.vue) 加载文档时，会先拿服务端文档，再读取本地草稿。如果本地草稿和服务端内容有差异，就恢复本地草稿，并提示用户“已恢复本地草稿”。编辑过程中监听标题和内容变化，先写本地草稿，再通过 800ms debounce 自动保存到服务端。

保存成功后，`doSave()` 会调用 `clearDocumentDraft(docId.value)` 清理草稿；删除文档时也会清理；如果草稿内容和服务端内容一致，也会清理。这个基础防丢能力已经具备。后续改进是加服务端草稿表、版本号、更新时间比较和冲突弹窗，解决多端编辑和版本冲突问题。

## 14. 文档同步到知识库如何保证一致性？失败了怎么办？

[addDocumentToKnowledgeBase](f:/AI-Knowledge-Platform/src/api/documents.ts) 会先确认文档和知识库都属于当前用户，然后把文档内容切片。接着 upsert `knowledge_documents` 桥表，记录文档与知识库的关联；再删除该文档在该知识库下的旧切片；最后批量插入新切片。

这样重复同步时不会堆积旧数据，而是用最新文档内容覆盖旧切片。它还兼容旧数据库结构：如果 `knowledge_chunks.file_id` 有 not null 约束，直接插入文档切片失败，会创建一个 synthetic `.md` 文件记录，再带着这个 fileId 重试插入切片。

不足是这些步骤目前在前端串行调用，不是数据库事务。中途失败可能造成桥表已更新但切片没有完全写入。更稳的方案是把同步逻辑封装到 RPC 或 Edge Function，在数据库事务里完成旧切片删除、新切片插入和桥表更新，保证要么全部成功，要么回滚。

## 15. 这个项目有哪些主要性能瓶颈？你会怎么优化？

第一，RAG 检索现在会从数据库拉取当前知识库最多 500 个切片到前端，再做向量或关键词检索。中小规模知识库可以接受，但数据量变大后会慢。优化方向是把检索下沉到数据库或服务端，比如 pgvector、全文索引、BM25、rerank。

第二，embedding 生成依赖外部 AI API，受延迟、限流和失败影响。虽然项目支持批量生成 embedding，但更适合放到后台任务队列中处理，配合重试、并发限制、失败状态记录和进度展示。

第三，SSE 流式回答会频繁更新响应式消息，长回答时可能造成渲染压力。可以对 chunk 更新做节流，或者只更新当前 assistant 消息节点，避免整个消息列表频繁重渲染。

第四，管理后台和列表页需要分页、索引、虚拟滚动和聚合优化。当前路由组件已经使用动态 import，能降低首屏包体压力，但项目真正的大头不是包体，而是检索链路、AI 请求耗时、数据库聚合和流式渲染。
