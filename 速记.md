**一页面试速记表**
| 维度 | 速记内容 |
|---|---|
| 项目定位 | AI 知识库平台：文档管理、知识库构建、RAG 问答、AI 写作、共享广场、管理后台 |
| 技术栈 | Vue3 + TypeScript + Vite + Pinia + Vue Router + Element Plus + Supabase + OpenAI 兼容 API + SSE + ECharts |
| 核心链路 | 登录鉴权 → 文档编辑 → 文本切片 → 写入知识库 → 检索相关片段 → 构建 Prompt → AI 流式回答 → 保存消息和引用 |
| 状态管理 | `userStore` 管登录态、角色；`aiConfigStore` 管用户模型配置 |
| 数据来源 | Supabase Auth、Postgres 表、RPC、localStorage 草稿 |
| AI 能力 | 普通生成、SSE 流式生成、AI 写作助手、知识增强问答、Embedding 工具 |
| RAG 实现 | `chunkText` 切片，`retrieveChunks` 关键词召回，`buildQaPrompt` 注入上下文 |
| 权限设计 | 前端路由守卫做体验拦截，真实安全依赖 Supabase RLS |
| 性能关注 | 切片检索、Embedding 串行、流式渲染频率、后台统计聚合、列表分页 |
| 安全关注 | AI Key 前端可见、管理员操作需后端权限、前端权限不能替代 RLS |
| 可讲主亮点 | “我做的是一条完整的内容资产化到 AI 问答的链路，不只是 CRUD。” |

**模块亮点**
| 模块 | 亮点讲解 |
|---|---|
| 登录与权限 | 使用 Supabase Auth 恢复会话，路由守卫根据登录态和 admin 角色控制访问，做到“体验层拦截 + 数据层权限兜底”。 |
| 文档编辑 | Markdown 文档支持编辑、保存、分享、本地草稿和同步知识库，文档既是内容资产，也是 RAG 数据源。 |
| 知识库 | 把文档或文件统一组织成知识库，通过切片、入库、QA 配置，为后续问答提供上下文。 |
| 文本切片 | 按段落、句子、长度做分层切片，保证片段尽量完整，同时避免 Prompt 过长。 |
| AI 问答 | 会话、消息、知识检索、Prompt 构建、SSE 流式输出、来源引用形成闭环。 |
| Prompt 构建 | 将用户问题、知识片段、回答风格、系统约束统一封装，降低模型幻觉并提升可控性。 |
| AI 写作助手 | 在文档编辑器中提供润色、扩写、总结、续写，把 AI 能力嵌入实际写作流程。 |
| 用户 AI 配置 | 支持用户自定义 Base URL、API Key、模型名，可以接入 OpenAI 兼容服务，降低平台绑定。 |
| 管理后台 | 提供用户、文档、文件、聊天记录和统计分析，体现平台治理和运营能力。 |
| 埋点统计 | 通过统一事件常量记录登录、文档、AI 调用等行为，为后台分析提供数据基础。 |

**面试问答模板**
| 功能 | 面试问题 | 答案模板 |
|---|---|---|
| 项目整体 | 这个项目最核心的业务价值是什么？ | 它把用户文档变成可检索的知识资产，再通过 RAG 和 AI 流式问答提供知识服务，核心价值是“内容沉淀 + 智能问答”。 |
| 技术选型 | 为什么用 Vue3 + TS？ | Vue3 适合组件化和响应式开发，TS 能约束 API 数据、业务模型和组件状态，减少字段错误和空值问题。 |
| 路由权限 | 登录校验怎么做？ | 启动时先恢复 Supabase session，路由守卫判断是否登录；admin 页面通过 route meta 和用户角色判断访问权限。 |
| 安全 | 前端路由守卫安全吗？ | 不完全安全。前端守卫只负责用户体验，真正安全要靠 Supabase RLS 和数据库权限策略。 |
| 文档编辑 | 如何避免用户编辑内容丢失？ | 使用 localStorage 保存本地草稿，按文档 id 存储标题、内容和保存时间；改进方向是服务端草稿和版本冲突提示。 |
| 知识库 | 文档如何进入知识库？ | 文档内容先通过切片函数拆成多个片段，再批量写入 knowledge_chunks，后续问答从这些片段中召回上下文。 |
| RAG | 你的 RAG 流程是什么？ | 资料切片入库，提问时召回相关片段，把片段和问题组成 Prompt，再交给模型生成回答并展示引用来源。 |
| 检索 | 当前怎么召回知识片段？ | 当前主流程使用关键词和中文 ngram 做轻量 TopK 检索，优点是部署简单；后续可融合 Embedding 向量检索和 rerank。 |
| 切片 | 为什么不直接把全文发给模型？ | 全文会超出上下文窗口，也会增加成本和噪声；切片后只注入相关内容，可以提高准确性和效率。 |
| SSE | 流式回答怎么实现？ | 使用 fetch 读取 ReadableStream，通过 TextDecoder 增量解码，维护 buffer，解析 SSE 的 `data:` 事件并实时更新 UI。 |
| AI 配置 | 为什么支持用户自定义模型配置？ | 这样平台不绑定单一供应商，用户可以接入 OpenAI 兼容模型服务；缺点是 API Key 安全需要服务端代理进一步增强。 |
| AI 写作 | AI 写作助手怎么设计？ | 把润色、扩写、总结、续写抽象成不同 Prompt 动作，复用同一 AI 生成接口，嵌入文档编辑流程。 |
| 管理后台 | 后台统计怎么实现？ | 前端调用管理 API 和 Supabase RPC 获取用户数、文档数、AI 调用趋势等，再用 ECharts 渲染可视化。 |
| 性能 | 项目主要性能瓶颈在哪里？ | 大知识库前端拉取切片检索会慢，Embedding 串行生成成本高，流式回答频繁更新会带来渲染压力，后台统计需要聚合优化。 |
| 改进 | 如果继续优化你会先做什么？ | 第一，把 AI Key 和模型请求迁移到服务端代理；第二，接入向量检索和 rerank；第三，增加分页、缓存、并发限流和错误恢复。 |

**一句话背诵版**
这个项目可以讲成：“我用 Vue3 和 Supabase 搭了一个 AI 知识库平台，核心是把文档切片入库，提问时召回相关片段构建 Prompt，再通过 SSE 流式生成可追溯回答，同时补齐了登录权限、AI 配置、共享、埋点和管理后台。”
