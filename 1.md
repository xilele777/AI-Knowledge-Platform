2. 你的文本切片规则是“按段落/句子”，那你怎么处理跨段落语义被切断的问题？  
比如一个定义在上一段，限制条件在下一段，切片后只召回其中一段会导致答案错误。你有没有做 overlap、父子 chunk、标题继承、metadata 补全？chunk size 为什么这么设？

5. 流式消息“完成后持久化到 Supabase”，如果流式过程中用户刷新页面、切换会话、或者连接断了，消息状态怎么处理？

8. 文档编辑里你做了“防抖自动保存、本地草稿恢复、当前文档切片同步”。如果用户连续编辑、自动保存、切片同步同时发生，如何避免旧版本覆盖新版本？

9. 你封装了埋点上报并用 ECharts 展示趋势。请你讲清楚一次 AI 调用从触发到后台图表展示的完整链路。

## 参考解答

**1. 动态切换问答模式怎么做？**

我这里不是简单让用户手动选，而是前端封装了一层 `QA mode` 判断逻辑。

如果问题里有明确实体、文件名、标题、日期、专有名词，比如“张三的结论是什么”“XX 文档里怎么说”，我优先走关键词 + 向量混合检索，因为关键词能保证精确命中，向量负责补语义相近内容。

如果问题更偏总结、归纳、开放式，比如“总结这个知识库的项目风险”，我会优先走向量检索，扩大 top-k，再把多个 chunk 拼成上下文。

如果用户问题很短，比如“怎么配置”“这个呢”，会结合当前会话上下文补全 query 后再检索。

切换不是绝对的，我更倾向于混合检索：关键词命中高时提高关键词权重，语义类问题提高 embedding 权重。最后按综合分排序，比如：

```ts
score = vectorScore * 0.7 + keywordScore * 0.3
```

阈值上，如果最高相似度低于设定值，就不强行回答，而是提示“知识库中没有找到足够相关的信息”。

**2. 文本切片怎么避免语义被切断？**

一开始只按段落切，确实会遇到上下文断裂的问题。后来我做了几个处理：

第一，chunk 不是纯按字符数硬切，而是优先按标题、段落、句号等语义边界切。

第二，对相邻 chunk 做 overlap，比如当前片段保留上一段末尾的一小部分内容，避免定义和解释被拆开。

第三，给 chunk 带 metadata，比如文档 id、标题、段落序号、上级标题。这样召回时不只是拿一段孤立文本，还能知道它属于哪个章节。

第四，问答时如果召回到某个 chunk，可以顺带取它前后相邻 chunk，补足上下文。

chunk size 我会控制在一个不会让 embedding 语义过散、也不会太碎的范围，比如 300 到 800 中文字左右，具体会看文档类型调整。

**3. 为什么前端切片，而不是后端切片？**

这个项目定位偏个人/团队知识库，前期为了降低后端复杂度和快速验证，所以把切片逻辑放在前端完成。前端负责读取文件、解析文本、切片、批量写入 Supabase。

但我知道这不是最理想的长期方案。前端切片的问题是：大文件会占内存、浏览器可能卡顿、失败恢复更复杂。

所以我做了几个稳定性处理：

文件读取后分批处理，不一次性提交所有 chunk。

批量写入时控制并发，比如每批 20 到 50 条。

每个 chunk 生成稳定 hash，避免重复入库。

入库失败时保留失败批次，允许重新提交。

上传状态分为 parsing、embedding、saving、done、failed，避免用户不知道卡在哪。

如果项目继续演进，我会把切片和 embedding 迁到后端或 Edge Function，让前端只负责上传和展示进度。

**4. SSE 流式数据怎么解析？**

我用的是 SSE，因为这个场景主要是服务端单向推 token，前端不需要高频双向通信。相比 WebSocket，SSE 更轻，天然支持事件流，也更适合 AI 回复这种场景。

前端解析时不会假设每次收到的数据都是完整 JSON。我的处理方式是维护一个 buffer：

收到 chunk 后先 append 到 buffer。

按换行或 SSE 的 `data:` 边界拆分。

如果 JSON parse 失败，说明数据可能没收完整，就继续等下一段。

收到 `[DONE]` 后标记生成结束。

如果中途断连，就把当前消息标记为 interrupted 或 failed。

UI 层不是每来一个 token 就疯狂更新 DOM，而是做节流更新，比如几十毫秒合并一次，避免长回复时渲染压力过大。

**5. 流式消息完成后才持久化，刷新怎么办？**

我这里的核心思路是：UI 里先维护临时消息状态，最终完成后再持久化到 Supabase。

消息状态大概有：

```ts
streaming
completed
failed
interrupted
```

用户发送问题时，用户消息可以先入库；AI 消息在前端生成临时 id，流式展示。生成完成后，把完整 answer、引用 chunk、耗时、token 信息一起写入 Supabase。

如果中途刷新，严格来说这条 AI 消息可能丢失。所以更稳的方案是边生成边缓存，比如写 localStorage 草稿，或者后端生成时同步记录 partial message。

面试里我会主动说：当前方案优点是写库简单、避免频繁请求；缺点是断连恢复一般。如果要做生产级，我会改成服务端维护 message 状态，前端只订阅流。

**6. 怎么降低 RAG 幻觉？**

我主要从检索和 prompt 两侧控制。

检索侧：

只把相似度超过阈值的 chunk 放进上下文。

控制 top-k，避免无关内容太多。

混合关键词检索，避免实体类问题向量召回跑偏。

对 chunk 保留来源信息，便于回答时引用。

prompt 侧：

明确要求模型只能基于提供的知识库内容回答。

如果上下文没有答案，要回答“未找到相关信息”，不能编造。

回答时尽量附带来源文档或片段标题。

还有一个关键点：不是所有问题都必须送给模型。如果召回结果为空或者分数明显低，就直接在前端提示用户知识库缺少相关资料，这比让模型硬答更安全。

**7. Pinia + 路由守卫控制 admin 安全吗？**

只靠前端路由守卫肯定不安全。前端守卫只能提升用户体验，不能作为真正权限边界。

我项目里的 Pinia 主要负责保存登录态、用户信息、角色信息，然后路由守卫根据 role 控制页面是否能进入。

但真正的数据权限应该在 Supabase 侧兜底，比如开启 RLS，也就是 Row Level Security。admin 表、埋点表、用户表这些敏感数据，必须由 Supabase policy 判断当前用户 id 和角色。

所以我会这样回答面试官：

前端守卫负责“不让普通用户看到 admin 入口和页面”。

Supabase RLS 负责“即使用户绕过前端直接请求，也拿不到数据”。

角色信息不能只信前端 store，应该来自后端表或 JWT claims，并且最终由数据库策略校验。

**8. 自动保存和切片同步如何避免旧数据覆盖新数据？**

这个问题核心是并发和版本。

我的处理方式是：

编辑器输入后防抖保存，比如 800ms 或 1000ms 后保存。

保存请求发出前记录当前文档版本或 `updated_at`。

如果用户继续编辑，旧请求返回时不能直接覆盖当前状态。

可以用 `AbortController` 取消上一次未完成请求。

也可以维护保存队列，保证按顺序提交。

切片同步不是每次输入都同步，而是在文档保存成功后，再触发当前文档的 chunk 重建。

为了避免旧 chunk 残留，我会先按 document_id 删除旧 chunk，再插入新 chunk，或者用 version 标记，只让最新 version 的 chunk 生效。

如果多人同时编辑，这套还不够，需要引入协同编辑或版本冲突提示。这个项目更偏个人/团队知识库基础共享，所以我会说明目前主要解决单用户编辑下的数据安全。

**9. AI 调用埋点到 ECharts 的完整链路？**

一次 AI 调用开始时，前端会上报一条事件，比如：

```ts
event_name: 'ai_chat_start'
user_id
session_id
doc_id
mode
timestamp
```

调用完成后再上报：

```ts
event_name: 'ai_chat_finish'
duration
success
error_type
retrieved_count
```

如果失败，就上报 `ai_chat_error`。

这些埋点写入 Supabase 的 analytics 表。后台页面查询时按时间聚合，比如按天统计 AI 调用次数、成功率、活跃用户数，然后交给 ECharts 展示折线图、柱状图。

PV 和 UV 的区别是：

PV 看事件次数。

UV 按 user_id 或 session_id 去重。

活跃用户一般按当天产生过登录、编辑、AI 调用等关键行为的用户去重。

为了避免重复点击，我会在前端对按钮加 loading 状态，并且埋点里带 request_id，同一个 request_id 重复上报时可以去重。

**10. 这个项目技术含量最高的点是什么？怎么量化？**

我会说技术含量最高的是 AI 问答链路，因为它不是单一接口调用，而是包含：

文档切片。

向量化。

检索召回。

上下文组装。

流式输出。

消息持久化。

错误恢复。

前端体验优化。

量化上可以这样说：

普通非流式接口要等完整回答生成后才展示，长回答可能 8 到 15 秒用户才看到内容；用了 SSE 后，首字出现时间可以控制在 1 秒左右，用户感知明显更好。

大文档入库方面，原来一次性写入容易失败；改成分批写入和失败重试后，几百个 chunk 的文档也能稳定入库。

请求优化方面，文档编辑从每次输入都可能触发保存，改成防抖自动保存，显著减少 Supabase 写请求。

如果面试官继续追问“有没有真实指标”，你可以诚实说：

这个项目没有做严格 A/B，但是我在开发阶段用大文档和长回复做过压测对比，主要关注首字时间、入库成功率、请求次数和失败恢复。后续如果生产化，我会把这些指标接入埋点系统持续监控。
